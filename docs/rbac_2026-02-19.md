# RBAC Setup (`USERS / GROUP_USER / PERMISSION_GROUP / FORM_CONTOL`)

Date: **2026-02-22**

This project now uses the RBAC schema requested in Oracle:

- `USERS`
- `GROUP_USER`
- `PERMISSION_GROUP`
- `FORM_CONTOL` (database typo of `form_control`, handled automatically in code)

## Authentication

- Login URL: `/login`
- Sign-up URL: `/signup`
- Logout URL: `POST /logout`
- Session key: `staff.auth`

Login/sign-up uses `USERS`:

- `USERS.E_ID` links user to `EMPLOYEES.EMPLOYEE_ID`
- `USERS.G_ID` links user role to `GROUP_USER.G_ID`
- `USERS.STATUS` controls active/inactive login

## RBAC Sources

- Roles: `GROUP_USER`
- Permission catalog: `FORM_CONTOL` (or `FORM_CONTROL` if present)
- Role-permission mapping: `PERMISSION_GROUP`

## Admin RBAC Pages

All are guarded by `staff.ability:dashboard.admin`:

- Dashboard: `/admin/dashboard`
- Users: `/admin/rbac/users`
- Roles: `/admin/rbac/roles`
- Permissions: `/admin/rbac/permissions`

## Notes

- Existing route ability checks still use permission codes like `products.read`, `orders.read`, etc.
- `StaffAuth::ensureRbacTablesSeeded()` seeds default roles/permissions/mappings.
- Code auto-fixes `USERS_FK1` to reference `USERS.E_ID -> EMPLOYEES.EMPLOYEE_ID`.
